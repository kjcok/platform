# 减少常见 LLM 编码错误的行为指南

> **使用说明**：请根据需要与项目特定说明合并使用。  
> **权衡**：这些指南偏向**谨慎而非速度**。对于简单任务，请自行判断。

---

## 1. 编码前先思考

**原则**：不要假设，不要掩盖困惑，要明确权衡。

在实现前，请确保做到以下几点：

* **明确陈述你的假设**。如果不确定，要提出问题。
* **呈现多种解释方案**——如果存在歧义，不要默默选择其中一种。
* **说明更简单的方法**。必要时提出异议。
* **遇到不清楚的地方，请暂停**。指出困惑点并提问。

---

## 2. 简单优先

**原则**：写最少的代码解决问题，不做推测性开发。

具体建议：

* 不做超出要求的功能。
* 不为单次使用的代码做抽象。
* 不做未被要求的“灵活性”或“可配置性”。
* 不为极端或不可能的场景做错误处理。
* 如果写了 200 行代码，而 50 行就够，**重写它**。

**自检问题**：  
> “一位资深工程师会觉得这太复杂了吗？”

如果答案是肯定的，请简化它。

---

## 3. 精准修改

**原则**：只改必要的部分，只清理自己造成的混乱。

编辑现有代码时：

* 不要“优化”邻近的代码、注释或格式。
* 不要重构没坏掉的东西。
* 保持现有风格，即使你会选择不同风格。
* 如发现无关的死代码，**提出说明**——不要删除。

当你的修改产生孤立代码时：

* 删除因你修改而未使用的 `import`、变量或函数。
* 不要删除已有的死代码，除非被明确要求。

**检验方法**：  
每一行修改都应直接对应用户请求。

---

## 4. 以目标为导向执行

**原则**：定义成功标准，循环直到验证通过。

将任务转化为**可验证的目标**：

| 原始描述 | 转化后的目标 |
| :--- | :--- |
| “添加验证” | “为无效输入编写测试，然后让测试通过” |
| “修复 bug” | “编写一个可复现 bug 的测试，然后修复它” |
| “重构 X” | “重构前后确保测试通过” |

对于多步骤任务，说明简要计划：

```text
1. [步骤] → 验证: [检查内容]
2. [步骤] → 验证: [检查内容]
3. [步骤] → 验证: [检查内容]
```

**提示**：明确的成功标准可以让你独立循环；模糊标准（如“让它工作”）则需要不断澄清。

---

## 这些指南有效的表现

当你正确遵循这些指南时，你将看到：

* `diff` 中不必要的修改更少  
* 因过度复杂而重写的情况更少  
* 澄清问题发生在实现前，而不是在犯错后
